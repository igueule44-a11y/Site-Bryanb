Le making-of d'un déploiement raté : 8 allers-retours pour pousser un fichier HTML

Pourquoi déployer un simple fichier HTML a pris huit allers-retours avec mon assistant IA, et les 4 erreurs concrètes qui ont fait perdre le fil.

Vendredi soir, 23h. Je veux pousser un simple fichier blog.html sur omnyx.agency. Spoiler : ça m'a pris 8 tentatives et 2h. Voici ce qui s'est passé.

Le contexte

J'ai une app React + Vite déployée sur Vercel. Le blog.html que je voulais ajouter n'était pas dans le code source — c'était une page statique standalone que j'avais préparée à part. Logique : je la mets dans public/, je push, Vercel la sert, terminé.

Les 4 erreurs que j'ai faites

Erreur 1 : supposer que public/ servait les pages statiques à la racine

Mon blog.html était à public/blog.html. Au build, Vite l'a copié dans dist/blog.html. Ça aurait dû marcher. Mais Vite, par défaut, sert l'index React sur toutes les routes qui n'ont pas d'extension de fichier spécifique. Le fichier était bien sur le serveur, mais la route /blog.html était interceptée par le fallback SPA.

Erreur 2 : ne pas avoir de vercel.json propre

Mon vercel.json n'avait pas de section cleanUrls, ni de rewrites explicites. Le routing SPA avait la priorité sur tout.

Erreur 3 : faire confiance à l'agent IA sans vérifier

Mon agent a passé 4 itérations à me dire "ah, c'est bon, j'ai compris, c'est un problème de path". À chaque fois, sa solution était plausible mais incorrecte. J'aurais dû tester la solution moi-même après chaque itération, pas après 4.

Erreur 4 : ne pas documenter le pattern

Au final, la solution était : transformer blog.html en une entrée multi-page dans vite.config.ts (rollupOptions.input), avec output en blog/blog-article.html pour chaque article. C'est un pattern que je vais réutiliser 50 fois. Je l'aurais vu tout de suite si j'avais eu un notes/patterns.md.

La vraie leçon

Un agent IA est fort pour générer des hypothèses, faible pour les valider. Mon process devrait être : l'agent propose, je teste, l'agent propose la suite. Pas l'agent propose 4 hypothèses en série pendant que je le regarde faire.

Ce que j'ai changé depuis

  • Vérification après chaque changement de config, pas après une série
  • Pattern documenté dans le repo pour les pages statiques multi-entry Vite
  • Leçon appliquée : ce blog utilise exactement le pattern que j'aurais dû appliquer dès le départ (pages HTML statiques servies comme telles, pas via le routing SPA)

Si tu construis un process où l'IA fait des changements de config ou d'archi, on peut t'aider à mettre le bon cadre.
